## 架构与关键实现点

### 控制连接 + 工作连接模型

```
Client ──控制连接──► Server
                        │  收到用户连接时
                        ▼
Client ◄─── ReqWorkConn ────
Client ─── NewWorkConn ────► Server
                              │
用户 ◄──── work conn ─────────┘
```

- **控制连接**：长连接，用于交换注册、心跳、新工作连接请求等控制消息
- **工作连接**：每次用户请求触发（或从连接池复用），用于透传实际数据

### TCP 多路复用（tcp_mux）

tcp_mux 允许一条工作连接承载多个并发用户连接，内部通过自定义帧协议区分不同虚拟流。

帧格式（`crates/arp-common/src/transport/mux.rs`）：

| 类型值 | 名称 | 含义 |
|--------|------|------|
| 1 | `FRAME_OPEN` | 新虚拟流建立，携带 stream_id |
| 2 | `FRAME_DATA` | 数据帧，携带 stream_id + payload |
| 3 | `FRAME_CLOSE` | 虚拟流关闭 |
| 4 | `FRAME_PING` | 心跳请求 |
| 5 | `FRAME_PONG` | 心跳响应 |

**重要**：tcp_mux 只对 `type = "tcp"` 的代理生效。`http`/`https` 代理不走 mux 路径，永远直接透传原始 HTTP 字节流。判断逻辑在 `crates/arp-client/src/control/mod.rs` 的 `register_proxies()` 中：

```rust
let multiplexer = if proxy_config.multiplexer.is_empty()
    && self.config.transport.tcp_mux
    && proxy_config.proxy_type == ProxyType::Tcp   // 只对 Tcp 类型生效
    && !has_lb_group
{
    "tcp_mux".to_string()
} else {
    proxy_config.multiplexer.clone()  // http/https 走这里，返回 ""
};
```

### HTTP/HTTPS VHost（虚拟主机路由）

服务端实现位于 `crates/arp-server/src/proxy/vhost.rs`：

- 服务端在 `vhost_http_port` 与 `vhost_https_port` 启动共享监听器
- 客户端注册 `type = "http"` 时，服务端记录域名 → 代理实例的映射
- HTTP 连接：读取请求头，解析 `Host` 后路由到对应代理
- HTTPS 连接：读取 TLS ClientHello，解析 `SNI` 后路由到对应代理
- 路由成功后通过 `request_work_conn` 请求客户端建立工作连接，将请求头写入工作连接后双向 copy

**注意事项**：
- `http`/`https` 代理必须配置 `custom_domains` 或 `subdomain`，否则注册失败
- 服务端 `vhost_http_port` 默认为 0（禁用），使用前必须显式配置
- `http`/`https` 客户端代理复用 TcpProxy 的本地转发逻辑，但 `tcp_mux = false`

### UDP 代理

- 服务端 `udp` 代理监听公网 UDP 端口
- UDP 报文通过持久工作连接复用传输（UDP mux），避免每包建连
- 客户端 `udp` 代理将报文发往本地 UDP 服务，并将响应报文回传
- UDP 数据链路支持可选 `use_compression` 与 `use_encryption`（AES-GCM）

### 连接池（pool_count）

- 客户端根据 `transport.pool_count` 预热工作连接，并在连接消费后自动补充
- 服务端在无待处理请求时缓存空闲工作连接，优先复用
- 目的：用户请求到来时无需等待新连接握手，降低延迟

### TCP/STCP 负载均衡

- 同 `remote_port` + `load_balancer.group/group_key` 的 `tcp/stcp` 代理共享一个服务端监听端口
- 服务端按轮询挑选分组后端，下发带 `proxy_name` 的 `ReqWorkConn`
- 客户端在 `ReqWorkConn(proxy_name)` 阶段依据健康状态决定是否创建工作连接
- 服务端对失败后端执行临时摘除（eject），窗口后自动恢复参与调度

### XTCP NAT 穿透

- `xtcp` provider 注册时不占用 `remote_port`，由 visitor 触发会话
- visitor 发起 `NatHoleVisitor`，服务端校验 `sk` 并转发 `NatHoleClient` 给 provider
- provider 创建临时 P2P 监听并回传 `NatHoleResp(client_addr)`，visitor 直连该地址后与 provider 本地服务透传
- 当前实现包含 NAT 打洞协商与 best-effort UDP 打洞包；在受限 NAT 环境下可能需要 `fallback_to_relay = true`

### WebSocket 传输

- 当 `transport.protocol = "websocket"` 时，控制连接与工作连接均通过 WS 建立
- 通过 `ws_stream` 桥接器把 WS 二进制帧映射为字节流，复用现有消息编解码与代理数据通道逻辑
- 当 `transport.tls.enable = true` 时，WS 握手运行在 TLS 之上，形成 `wss` 连接

### WSS 与 QUIC 的实现差异

- `wss`：复用已有 TCP/TLS 字节流路径，在应用层增加 WebSocket 握手与二进制帧桥接，偏"兼容性增强"
- `quic`：使用独立的 QUIC endpoint 与双向流，传输路径和握手模型与 TCP/TLS/WS 分支完全不同，偏"传输能力增强"

### 管理接口

- 服务端在 `dashboard_port > 0` 时启动 Dashboard HTTP 服务
- `GET /healthz`：健康检查
- `GET /metrics`：Prometheus 指标（控制连接数、代理数、工作连接数、TCP 字节数、tcp_mux 流计数等）
- `GET /api/v1/status`：服务端 JSON 状态
- `GET /api/v1/proxies`：当前代理键列表
- `GET /api/v1/events/stream`：SSE 实时事件流

### HMAC 认证实现

- 客户端发送 `privilege_key = HMAC-SHA256(token, timestamp)`
- 服务端对所有已知 Token 逐一验签
- 强制 300 秒（5 分钟）防重放窗口，超出范围的时间戳直接拒绝
- v0.4.0 之前的版本不支持此机制，新旧版本不兼容

## 代码结构（关键路径）

```
crates/
├── arp-common/
│   ├── src/config/mod.rs          # 配置结构体（ClientConfig、ServerConfig、ProxyConfig 等）
│   ├── src/transport/mod.rs       # MessageTransport 抽象，原始流切换时保留预读缓冲
│   ├── src/transport/mux.rs       # TCP mux 帧读写
│   └── src/message/mod.rs         # 控制协议消息定义
├── arp-server/
│   ├── src/proxy/mod.rs            # 服务端代理管理（注册、分发 ReqWorkConn）
│   ├── src/proxy/vhost.rs          # HTTP/HTTPS vhost 监听与路由
│   ├── src/proxy/tcp.rs            # TCP 代理监听与 accept
│   └── src/control/mod.rs          # 服务端控制连接处理
└── arp-client/
    ├── src/control/mod.rs           # 客户端控制连接、register_proxies、work conn 分发
    ├── src/proxy/mod.rs             # 代理类型选择（根据 type 分配 TcpProxy/UdpProxy 等）
    └── src/proxy/tcp.rs             # TcpProxy：tcp_mux / stcp / raw copy 三条路径
```

### 关键实现细节

- `MessageTransport` 切换到原始 `TcpStream` 时必须处理 codec 预读缓冲（`buffered_bytes`），否则可能丢失 `StartWorkConn` 后的首包数据
- `http`/`https` 客户端代理当前复用 TCP 本地转发逻辑（`TcpProxy`），但 `tcp_mux` 强制为 false
- 服务端 vhost listener 接收到用户连接后，先读 HTTP 请求头或 TLS ClientHello，解析路由 key 后再请求 work conn，然后将已读的请求头写回 work conn（保证客户端收到完整 HTTP 请求）

## 新增功能开发指引

### 新增代理类型

1. 在 `arp-common/src/config/mod.rs` 的 `ProxyType` 枚举中添加新类型
2. 在 `arp-client/src/proxy/mod.rs` 中根据新类型分配对应的 Proxy 实现
3. 在 `arp-server/src/proxy/mod.rs` 中添加服务端注册与工作连接分发逻辑
4. 更新 `examples/` 目录下的示例配置
5. 更新 `docs/配置指南.md` 和 `README-zh.md`

### 新增传输协议

1. 在 `arp-common/src/config/mod.rs` 的 `TransportProtocol` 枚举中添加新协议
2. 在 `arp-common/src/transport/mod.rs` 中实现新传输层的连接建立逻辑
3. 服务端与客户端在初始化时根据协议类型选择对应传输实现
4. 需要处理 UDP/TCP 的差异（连接 vs 无连接）

### 修改配置字段

- 所有配置结构体使用 `#[serde(default)]` 保证向后兼容性
- 新增必填字段会破坏现有配置文件，应提供合理默认值
- 默认值可以通过 `#[serde(default = "fn_name")]` + 独立函数实现

## 常见开发陷阱

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| StartWorkConn 后首包丢失 | MessageTransport 有 codec 预读缓冲未处理 | 切换到原始流时先消费 `buffered_bytes` |
| http 代理触发 mux frame 错误 | 旧版二进制或旧进程污染日志 | 清理旧进程，确认使用新编译的二进制 |
| vhost 注册成功但访问空响应 | `vhost_http_port` 端口被其他进程占用 | `ss -tlnp` 检查端口占用，改用其他端口 |
| TLS 握手失败 | `server_name` 与证书 SAN 不匹配 | 检查证书 SAN，确保包含 `server_addr` 对应的域名/IP |
| 代理注册被拒绝（名称冲突） | 不同 `client_id` 使用了相同 `proxy.name` | 为每个节点分配唯一的 `client_id` |
| 工作连接 auth 失败 | Token 不一致或版本不兼容（v0.4.0+ HMAC） | 检查双端 Token 一致，确认版本兼容 |
